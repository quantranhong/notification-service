1. Overview & scope

The Notification Service delivers messages through SMS, Email, and Push Notifications, targeting internal users (operators, customer services) and mobile application users. It is designed for high availability (minimum two active instances) and uses PaaS for delivery (e.g. AWS SES/SNS, Firebase FCM).

Key capabilities:

  • Delivery channels: SMS, Email, Push
  • Audiences: INTERNAL, MOBILE_APP
  • Immediate and scheduled notifications
  • REST and GraphQL APIs; OAuth2 + JWT
  • Separate Scheduler Service (Quarkus) for cron-driven processing

Traffic reaches the Notification Service via a load balancer (HTTPS). The Scheduler Service calls the internal endpoint to process due scheduled notifications. The Notification Service uses PostgreSQL for core data, MongoDB for event logs, and Redis for idempotency and rate limiting. Delivery is delegated to PaaS (SES, SNS, FCM).

3. Component detail

3.1 Notification Service (Spring Boot)

ComponentResponsibility
REST APISend notification, create/list/cancel scheduled, subscriptions, templates, internal process-due-scheduled
GraphQLMutations: sendNotification, createScheduledNotification, cancelScheduledNotification; queries: scheduledNotifications, subscriptions
OAuth2 / JWTValidate tokens for REST/GraphQL; internal endpoint optionally protected by API key
Notification EngineResolve recipients, render templates, call channel providers, write audit events, idempotency
SMS / Email / Push providersAbstractions over AWS SNS/SES, Firebase FCM (or stubs in dev)

3.2 Scheduler Service (Quarkus)

Single responsibility: run a cron (default every minute), call POST /api/v1/internal/process-due-scheduled on the Notification Service with optional X-Internal-Api-Key. No direct database access.

3.3 Data stores

StorePurpose
PostgreSQLTemplates, subscriptions, internal users, scheduled notifications
MongoDBAppend-only notification events (audit, analytics)
RedisIdempotency keys (TTL), rate-limit counters

4. Data stores & models

4.1 PostgreSQL (core domain)

EntityKey fields
notification_templateid, channel (SMS/EMAIL/PUSH), name, body_template, subject_template, external_template_id
notification_subscriptionid, user_id, channel, audience_type (INTERNAL/MOBILE_APP), destination (phone/email/token)
internal_userid, email, roles (e.g. operators, support)
scheduled_notificationid, template_id, channel, audience_type, audience_filter, scheduled_at, timezone, status (PENDING/SENT/CANCELLED), context_json

4.2 MongoDB (events)

notification_events: request_id, channel, recipient, payload, status, sent_at, provider_response (append-only audit log).

4.3 Redis

Keys for idempotency (per request/key with TTL); optional rate-limit counters.

5. Interaction: Send immediate notification

sequenceDiagram
    participant Client
    participant API as REST / GraphQL
    participant Engine as Notification Engine
    participant PG as PostgreSQL
    participant Redis as Redis
    participant Channel as Channel Provider
    participant MDB as MongoDB

    Client->>+API: POST /api/v1/notifications
    API->>+Engine: sendToAudience(...)
    Engine->>PG: Resolve recipients
    PG-->>Engine: destinations
    loop each recipient
        Engine->>Redis: idempotency check
        Engine->>Channel: send(...)
        Channel-->>Engine: DeliveryResult
        Engine->>MDB: NotificationEvent
    end
    Engine-->>-API: results
    API-->>-Client: 200 OK

7. Interaction: Scheduled notifications (Scheduler Service)

sequenceDiagram
    participant Cron
    participant Scheduler as Scheduler Service
    participant NS as Notification Service
    participant Internal as InternalProcessController
    participant SchedSvc as ScheduledNotificationService
    participant Engine as Notification Engine
    participant PG as PostgreSQL
    participant Channel as Channel Provider
    participant MDB as MongoDB

    Cron->>+Scheduler: Trigger (every minute)
    Scheduler->>NS: POST /internal/process-due-scheduled
    NS->>+Internal: processDueScheduled()
    Internal->>+SchedSvc: processDueScheduled()
    SchedSvc->>PG: findDueScheduled(PENDING, now)
    PG-->>SchedSvc: due list
    loop each due
        SchedSvc->>PG: status = SENT
        SchedSvc->>+Engine: sendToAudience(...)
        Engine->>Channel: send
        Engine->>MDB: event
        Engine-->>-SchedSvc: done
    end
    SchedSvc-->>-Internal: done
    Internal-->>-NS: 202
    NS-->>-Scheduler: 202

10. Deployment & scalability

  • Notification Service: Stateless; run ≥2 instances behind a load balancer (e.g. AWS ALB, Azure App Gateway). TLS 1.2+.
  • Scheduler Service: Run one instance (or use a distributed lock if multiple).
  • PostgreSQL: Primary + read replicas; connection pooling (HikariCP).
  • MongoDB: Replica set for event log.
  • Redis: Cluster or managed (ElastiCache / Azure Cache).
  • Secrets: AWS Secrets Manager / Azure Key Vault; no secrets in code.
